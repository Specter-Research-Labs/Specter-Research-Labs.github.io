Log File Schemas

For operational jq/inspection workflows, see docs/ops/log-query-cookbook.md.

Each corpus run creates a timestamped directory under logs/ with the following structure:

logs/corpus-YYYY-MM-DD-HHMMSS/
├── summary.json.gz           # Run-level aggregates
├── goal_cache.json.gz        # Goal signature mappings
├── corpus.log                # Runtime log
├── run_config.json           # Run config snapshot
├── run_status.json           # Run status (completed/failed/partial)
├── sheaf_analysis.json       # Sheaf metrics (generated by experiments/cellular_sheaf)
├── failure_analysis.json     # Failure patterns (if --analysis)
├── analysis_report.json      # Intervention classification (if --analysis)
├── all_ged_matrices.json     # Cross-theorem GED data
├── root_goal_similarity.json # Cross-theorem root-goal similarity (postprocess)
├── postprocess_metrics.json  # Postprocess report + params/provenance (postprocess)
├── external_statement_similarity.json # External backends: problem statement similarity (postprocess)
├── report.md                 # Human-readable summary
└── <theorem_name>/           # Per-theorem directory
    ├── wild_type_*.json(.gz)
    ├── <intervention>_*.json(.gz)
    ├── wild_type_metrics.json
    ├── <intervention>_metrics.json
    ├── <intervention>_comparison.json
    ├── wild_type_mcts_tree.json
    ├── <intervention>_mcts_tree.json
    ├── wild_type_mcts_trace.jsonl (if --trace-mcts)
    ├── <intervention>_mcts_trace.jsonl (if --trace-mcts)
    ├── ged_matrix.json
    ├── attractor_clusters.json (if >=2 variants)
    ├── basin_analysis.json (basin mode only)
    └── *_search_trace_graph.json (optional)

Note on postprocess-only metrics: Some metrics are computed in postprocess and written back into summary.json.gz and *_comparison.json without rerunning the solver. k_search_efficiency is one of these and requires *_mcts_trace.jsonl for its primary null model.

Multi-provider runs use a top-level directory with per-provider subdirs:

logs/corpus-YYYY-MM-DD-HHMMSS/
├── providers_summary.json          # Provider-level aggregates
├── providers_theorem_summary.json  # Per-theorem cross-provider view
├── providers_report.md             # Human-readable multi-provider summary
└── provider=<name>/                # Full single-provider run layout

Backend note: External backends (Coq/ATP/SMT) use the same run-level run_config.json / run_status.json / summary.json.gz schemas, but may omit Lean-only artifacts (e.g., goal_cache.json.gz, *_history.json, *_assembly.json.gz). Use run_status.json capability flags and file presence checks to gate backend-specific analysis (e.g., sheaf metrics).

Agent-Mode CLI Output

wonton.py supports --agent, which emits one JSON object per line for structured consumption by orchestration systems. Lean runs also force --plain to suppress interactive UI output.

Lean --agent events:

{"event": "start", "backend": "lean", "provider": "reprover", "providers": null, "run_id": "corpus-YYYY-MM-DD-HHMMSS", "log_dir": "..."}
{"event": "end", "status": "completed", "backend": "lean", "run_id": "corpus-YYYY-MM-DD-HHMMSS", "log_dir": "...", "summary_path": ".../summary.json.gz", "postprocess_command": "...", "errors": []}

External backends (e, vampire, z3, coq) emit the same start/end shape without run_id and provider fields.

Run-Level Files

run_status.json

Purpose: Marks whether a run completed successfully and records minimal metadata to filter partial runs in analysis/viz.

Structure:

{
  "status": "completed",
  "started_at": "2026-01-26T12:30:36",
  "completed_at": "2026-01-26T13:18:02",
  "goal_id_scheme": "checkpoint",
  "partial_results": false,
  "capabilities": {
    "has_proof_term": true,
    "has_proof_term_pretty": true,
    "has_assembly_trace": true,
    "has_proof_term_metrics": true
  }
}

Notes:

  • status is running, completed, or failed.
  • partial_results is true if the run crashed but wrote partial outputs.
  • capabilities declares which optional artifacts are expected to exist for the run.
  • External runs set goal_id_scheme to external (Lean uses checkpoint).
  • Failed runs may include error and traceback fields.

run_config.json

Purpose: Snapshot of run inputs (backend/provider choice, corpus selection, budgets, flags) for reproducibility.

Structure (minimal):

{
  "format_version": 2,
  "run_id": "corpus-2026-02-04-143736",
  "log_dir": "/.../dossiers/wonton-soup/logs/corpus-2026-02-04-143736",
  "created_at": "2026-02-04T14:37:36",
  "provider": "heuristic",
  "mode": "dev",
  "corpus": "easy"
}

Notes:

  • format_version tracks log schema versions (current: 2).
  • Single-provider runs include provider, provider_label, and provider_desc.
  • Multi-provider runs include providers, providers_meta, and multi_provider=true; each provider=<name>/ subdir has its own single-provider run_config.json.

summary.json.gz

Purpose: Complete results for all theorems in a single file.

Structure:

{
  "run_id": "corpus-2026-01-07-120200",
  "crashed": [],
  "goal_sig_scheme": "ast",
  "goal_sig_stats": {"ast_missing": 0},
  "theorems": [
    {
      "name": "or_left",
      "wild_type": {
        "solved": true,
        "iterations": 2,
        "proof_term_hash": "abc123...",
        "k_search_efficiency": { ... },
        "metrics": { ... }
      },
      "interventions": [
        {
          "name": "block_left",
          "is_control": false,
          "baseline_solved": true,
          "blocked": ["left"],
          "solved": true,
          "status": "solved",
          "ged_search_graph": {
            "value": 2.0,
            "normalized": 0.12,
            "valid": true,
            "validity_notes": [],
            "trace_source": "mcts",
            "trace_completeness": "full"
          },
          "ged_proof_graph": null,
          "ged_trace_graph": null,
          "k_search_efficiency": { ... },
          "metrics": { ... }
        }
      ]
    }
  ],
  "aggregates": {
    "theorem_count": 40,
    "wild_type_solve_rate": 0.7,
    "intervention_count": 214,
    "intervention_solve_rate": 0.345,
    "crashed_count": 0,
    "ged_validity": {
      "ged_search_graph": {"valid": 210, "invalid": 4},
      "ged_proof_graph": {"valid": 0, "invalid": 0},
      "ged_trace_graph": {"valid": 0, "invalid": 0}

    "goal_type_tactic_matrix": { ... }
  }
}

Reading:

gzcat logs/.../summary.json.gz | jq '.theorems | length'
gzcat logs/.../summary.json.gz | jq '.aggregates'

Notes:

  • ged_* fields are optional and may be null when the backend cannot produce that family.
  • trace_source/trace_completeness are required when ged_trace_graph is present.

goal_cache.json.gz

Purpose: Maps goal states to signatures and tracks tactic outcomes per signature.

Note: mvar_id values in logs are checkpoint-scoped goal IDs of the form cp<id>:<lean_mvar_id> (raw Lean mvar_ids can repeat across REPL rollbacks).

Structure:

{
  "sig_scheme": "ast",
  "sig_stats": {
    "ast_missing": 0
  },
  "mvar_to_sig": {
    "cp1:_uniq.560": "sig_abc123",
    "cp1:_uniq.561": "sig_def456"
  },
  "entries": {
    "sig_abc123": {
      "sig": "sig_abc123",
      "type_expr": { ... },
      "hyp_exprs": [ ... ],
      "occurrences": {
        "cp1:_uniq.560": {
          "mvar_id": "cp1:_uniq.560",
          "outcomes": {
            "0": [true, false, true],
            "3": [false, false]
          }
        }
      }
    }
  }
}

Key fields:

  • mvar_to_sig: Maps metavariable IDs to signature hashes
  • entries[sig].occurrences[mvar_id].outcomes: Tactic success/failure by family index
  • Family indices: 0=simplify, 1=rewrite, 2=intro, 3=split, 4=cases, 5=closer, 6=contradiction, 7=arith, 8=automation, 9=other

Use case: Analyze which goal signatures respond consistently to which tactic families.

root_goal_similarity.json

Purpose: Cross-theorem similarity over each theorem’s root goal(s), using normalized goal-AST ordered tree edit distance (TED).

Structure:

{
  "schema_version": 1,
  "valid": true,
  "validity_notes": [],
  "goal_sig_scheme": "ast",
  "distance_metric": "normalized_goal_ast_ted",
  "distance_threshold": 0.2,
  "theorems": ["thm_a", "thm_b"],
  "distance_matrix": {
    "thm_a": {"thm_a": 0.0, "thm_b": 0.31},
    "thm_b": {"thm_a": 0.31, "thm_b": 0.0}
  },
  "clusters": { "...": "see analysis/attractors.py AttractorAnalysis.serialize()" }
}

Notes:

  • Requires goal_sig_scheme=ast (it uses goal_cache.json.gz AST payloads).
  • Produced by wonton.py postprocess.
  • May be skipped (valid=false) when the run contains too many theorems for a full NxN matrix (see postprocess_metrics.json params).

postprocess_metrics.json

Purpose: Provenance + parameter snapshot for post-run heavy metrics.

Structure:

{
  "schema_version": 1,
  "valid": true,
  "computed_at": "2026-02-07T02:31:56+00:00",
  "params": {
    "max_soft_ged_nodes": 60,
    "max_soft_ged_edges": 120,
    "max_novelty_pairs": 200000,
    "max_path_dp_cells": 2000000,
    "max_root_goal_theorems": 400
  },
  "inputs": {
    "summary_sha256": "…",
    "goal_cache_sha256": "…"
  },
  "runs": [
    {"run_dir": "...", "provider": null, "goal_sig_scheme": "ast", "updated_interventions": 4}
  ]
}

Notes:

  • Written by wonton.py postprocess.
  • The postprocess step updates *_comparison.json and summary.json.gz in-place.

external_statement_similarity.json

Purpose: Cross-problem similarity over external backend inputs (TPTP/SMT-LIB), using normalized ordered tree edit distance (TED) over parsed statements.

Structure (dense or sparse):

{
  "schema_version": 1,
  "valid": true,
  "validity_notes": [],
  "corpus": "tptp",
  "root": "/path/to/tptp",
  "distance_metric": "normalized_ordered_ted",
  "distance_threshold": 0.25,
  "matrix_mode": "full",
  "matrix_kind": "dense",
  "problem_count_total": 200,
  "problems": ["PUZ001", "PUZ002"],
  "distance_matrix": {"PUZ001": {"PUZ001": 0.0, "PUZ002": 0.31}, "PUZ002": {"PUZ001": 0.31, "PUZ002": 0.0}},
  "clusters": { "...": "see analysis/attractors.py AttractorAnalysis.serialize()" }
}

Notes:

  • Written by wonton.py postprocess for runs with run_config.json: {\"mode\": \"external\"}.
  • Uses matrix_mode=auto|full|knn|sample with deterministic sampling/limits to avoid O(N^2) on large corpora.

sheaf_analysis.json

Purpose: Sheaf-theoretic consistency metrics. Generated by: python -m experiments.cellular_sheaf.run <corpus_dir>

Structure:

{
  "unique_sigs": 188,
  "total_mvar_ids": 1341,
  "collapse_ratio": 7.13,
  "equiv_consistency": 0.992,
  "tactic_residual": 6.92,
  "per_family_residual": {
    "0": 0.002,
    "1": 1.24,
    "4": 27.7
  },
  "intervention_deltas": {
    "block_rw": 20.8,
    "block_simp": -3.0
  }
}

Interpretation:

  • equiv_consistency: 0-1, how consistently same-signature goals respond to same tactics (1.0 = perfect)
  • tactic_residual: Average squared error when predicting child features from parent features via linear transform
  • intervention_deltas: How much blocking a tactic changes the residual (positive = more unpredictable)

failure_analysis.json

Purpose: Categorize why proofs failed.

Structure:

{
  "total_failures": 12,
  "by_pattern": {
    "TACTIC_DESERT": 4,
    "PARTIAL_PROGRESS": 6,
    "MODEL_HALLUCINATION": 2
  },
  "failures": [
    {
      "name": "theorem_name",
      "goal_type": "P -> Q",
      "failure_pattern": "TACTIC_DESERT",
      "total_iterations": 10,
      "failure_ratio": 1.0,
      "unique_tactics_tried": 3,
      "sample_tactics": ["intro", "apply h", "exact"],
      "technique_hint": "intro_elim"
    }
  ]
}

Failure patterns:

  • TACTIC_DESERT: All tactics fail, no progress possible
  • PARTIAL_PROGRESS: Some tactics work but proof doesn’t complete
  • MODEL_HALLUCINATION: Model generates invalid/malformed tactics

Per-Theorem Files

Each theorem gets its own subdirectory with wild-type and intervention results.

*_graph.json

Purpose: Graph artifact. In Lean runs this is a search graph (MCTS ProofGraph). External Coq/ATP/SMT runs emit proof graphs. The graph_kind field disambiguates (search_graph vs proof_graph).

Structure:

{
  "graph_kind": "search_graph",
  "nodes": [
    {"id": "cp1:_uniq.560", "type": "P -> Q", "solved": true}
  ],
  "edges": [
    {"source": "cp1:_uniq.560", "target": "cp1:_uniq.561", "tactic": "intro h"}
  ],
  "root": "cp1:_uniq.560"
}

Use case: Visualize proof search tree, compute graph metrics.

*_search_trace_graph.json

Purpose: Trace-derived graph when a true search graph is unavailable. Used for ged_trace_graph.

Structure:

{
  "graph_kind": "trace_graph",
  "trace_source": "tstp",
  "trace_completeness": "proxy",
  "valid": true,
  "validity_notes": [],
  "nodes": [
    {"id": "c1", "goal_sig": "sig_abc123", "node_kind": "clause", "in_proof": true}
  ],
  "edges": [
    {"src": "c1", "dst": "c2", "action_family": "Resolution", "action_norm": "res", "in_proof": true}
  ]
}

Use case: Compare trace-derived structure across variants without claiming true search equivalence.

*_history.json

Purpose: Detailed trace of every iteration.

Structure:

{
  "theorem": "theorem foo : P -> Q := by\n  sorry",
  "blocked_tactics": [],
  "iteration_count": 1,
  "detour_metrics": { ... },
  "solution_path": [
    {"goal": "P -> Q", "tactic": "intro h", "mvar_id": "cp1:_uniq.560"}
  ],
  "iterations": [
    {
      "iteration": 0,
      "selected_path": ["cp1:_uniq.560"],
      "attempts": [
        {
          "tactic": "intro h",
          "outcome": "success",
          "child_mvar_ids": ["cp1:_uniq.561"],
          "timestamp_ms": 45,
          "tactic_norm": "intro h",
          "goal_sig": "abc123...",
          "goal_sig_strict": "def456...",
          "goal_type": "P -> Q",
          "provider_id": "reprover"
        }
      ],
      "backprop_success": true,
      "terminal_reached": false
    }
  ]
}

Use case: Replay proof search, analyze tactic selection patterns.

*_assembly.json.gz

Purpose: Step-by-step proof term construction.

Structure:

{
  "theorem": "theorem foo : P -> Q := by\n  sorry",
  "rootMvarId": "cp1:_uniq.560",
  "steps": [
    {
      "tactic": "intro h",
      "mvarId": "cp1:_uniq.560",
      "partialTermBefore": null,
      "partialTermAfter": { "proofTerm": { ... } },
      "goalsClosed": [],
      "goalsOpened": ["cp1:_uniq.561"]
    }
  ],
  "finalTerm": {
    "rootId": "n0",
    "nodes": [
      ["n0", {"kind": "lam", "binderName": "h", "body": "n1", ...}],
      ["n1", {"kind": "app", "fn": "n2", "arg": "n3", ...}]
    ]
  }
}

Key fields:

  • steps: Each tactic application with before/after proof term state
  • finalTerm: Complete proof as expression DAG
  • nodes: List of [id, data] pairs representing Lean Expr nodes

Node kinds: lam (lambda), app (application), const (constant), bvar (bound variable), fvar (free variable), mvar (metavariable/hole), sort (type/Prop), lit (literal), proj (projection)

Use case: Animate proof construction, compare proof structures, debug tactic effects.

*_proof_term.json.gz

Purpose: Just the final proof term (subset of assembly).

Structure:

{
  "rootId": "n0",
  "nodes": [
    ["n0", {"kind": "lam", ...}],
    ...
  ]
}

Use case: Compare proof term structure without replay data.

*_metrics.json

Purpose: Computed statistics for a theorem variant (wild-type or intervention).

This file lives under the per-theorem directory (e.g. theorem_name/wild_type_metrics.json and theorem_name/block_simp_metrics.json).

Structure:

{
  "root_goal_sigs": ["sig_abc123", "sig_def456"],
  "trajectory": {
    "total_iterations": 5,
    "backtrack_count": 1,
    "max_depth_reached": 3,
    "unique_goals_visited": 4,
    "tactic_diversity": 3
  },
  "detour": {
    "total_attempts": 8,
    "success_count": 5,
    "failure_count": 3,
    "failure_ratio": 0.375
  },
  "proof_term": {
    "node_count": 45,
    "depth": 12,
    "width": 3
  },
  "solution_path": ["intro h", "apply h", "exact hp"],
  "tactic_fingerprint": "intro|apply|exact",
  "proof_term_pretty": "(fun h => ...)"
}

Notes:

  • root_goal_sigs are the goal signatures of root nodes in the serialized search graph (if present). They provide a cheap “root-goal fingerprint” for theorem-level comparisons.
  • tactic_fingerprint is currently written for solved wild-type variants; intervention variants may omit it.
  • proof_term_pretty is a best-effort Lean pretty-print of the proof term (when available). Treat it as display-only: it may contain internal Lean identifiers and unicode, and it is not guaranteed to round-trip as Lean syntax.

Purpose: Compare intervention to wild-type.

Structure:

  "blocked": ["simp"],
  "solved": true,
  "status": "solved",

  "hash_mismatch": false,
  "wild_type_axioms": {"all": ["Nat.add_comm", "..."], "counts": {"Nat.add_comm": 1}},
  "intervention_axioms": {"all": ["Nat.add_comm", "..."], "counts": {"Nat.add_comm": 1}},
  "ged_search_graph": {
    "value": 4.0,
    "normalized": 0.1,
    "valid": true,
    "validity_notes": [],
    "trace_source": "mcts",
    "trace_completeness": "full"
  },
  "ged_search_graph_soft": {
    "value": 3.2,
    "normalized": 0.08,
    "valid": true,
    "validity_notes": [],
    "trace_source": "mcts",
    "trace_completeness": "full"
  },
  "ged_proof_graph": null,
  "ged_trace_graph": null,
  "proof_term_diff": {
    "added_kinds": ["app"],
    "removed_kinds": []
  },
  "goal_novelty": {
    "valid": true,
    "validity_notes": [],
    "novel_goal_count": 2,
    "dropped_goal_count": 1,
    "novel_min_distance": {"count": 2, "mean": 0.12, "p50": 0.1, "p90": 0.14, "min": 0.1, "max": 0.14},
    "dropped_min_distance": {"count": 1, "mean": 0.0, "p50": 0.0, "p90": 0.0, "min": 0.0, "max": 0.0}
  },
  "solution_path_soft_distance": {
    "value": 0.2,
    "valid": true,
    "validity_notes": [],
    "wild_len": 7,
    "intervention_len": 8,
    "dp_cells": 56
  },
  "axiom_delta": ["Nat.add_comm"],
  "axiom_removed": [],
  "trajectory_diff": {
    "iteration_diff": 2,
    "backtrack_diff": 0
  },
  "trajectory_comparison": null
}

Key fields:

  • ged_search_graph / ged_proof_graph / ged_trace_graph: Family-specific GED objects
  • ged_search_graph_soft: Search-graph GED with a soft goal-AST substitution cost (requires goal_sig_scheme=ast)
  • goal_novelty: Novel/dropped goals under the intervention, summarized by min goal-AST TED to the other variant
  • solution_path_soft_distance: Soft edit distance between wild/intervention solution goal sequences (solved-only)
  • wild_type_axioms / intervention_axioms: Per-variant axiom fingerprints (when proof terms are available)
  • axiom_delta: Lemmas used in intervention but not wild-type
  • axiom_removed: Lemmas used in wild-type but not intervention

Postprocess note: ged_search_graph_soft, goal_novelty, and solution_path_soft_distance are filled by wonton.py postprocess. In-run payloads may contain placeholders (valid=false with a not computed in-run note) until postprocess runs.

Minimal Comparison Schema (Backend-Agnostic)

Purpose: Standardize per-intervention comparisons across backends and file locations (summary.json.gz and *_comparison.json).

Fields (required unless marked optional):

  • name (intervention name)
  • blocked (list of blocked tactics; optional)
  • solved (bool)
  • status (solved | failed)
  • ged_search_graph (object or null)
  • ged_search_graph_soft (object or null; optional)
  • ged_proof_graph (object or null)
  • ged_trace_graph (object or null)
  • wild_type_hash (string; optional)
  • intervention_hash (string; optional)
  • hash_mismatch (bool; optional)
  • wild_type_axioms / intervention_axioms (object; optional)
  • proof_term_diff (object; optional)
  • axiom_delta (list; optional)
  • axiom_removed (list; optional)
  • goal_novelty (object; optional)
  • solution_path_soft_distance (object; optional)
  • trajectory_diff (object; optional)
  • trajectory_comparison (object; optional)
  • metrics (object; optional)

Lean mapping notes:

  • Legacy ged maps to ged_search_graph.value.

ged_matrix.json

Purpose: Pairwise GED between all variants of this theorem. Lean uses search graphs; external runs use proof graphs.

Structure:

{
  "theorem": "or_left",
  "variants": ["wild_type", "block_intro", "block_simp"],
  "ged_matrix": {
    "wild_type": {
      "block_intro": 4.0,
      "block_simp": 0.0
    },
    "block_intro": {
      "wild_type": 4.0,
      "block_simp": 4.0
    },
    "block_simp": {
      "wild_type": 0.0,
      "block_intro": 4.0
    }
  }
}

Backend Artifact Mapping (Planning)

This table defines the intended mapping between backends and artifacts. Flags are indicative; confirm with the backend version.

Backend Search Graph Proof Graph Trace Graph Required Flags / Defaults trace_source
Lean *_graph.json optional proof subgraph n/a n/a mcts
Coq n/a (external) Coq proof DAG → proof graph n/a SerAPI n/a
E prover n/a TSTP proof object TSTP protocol (trace_graph) --auto --output-level=2 --tstp-out tstp
Vampire n/a TSTP proof object n/a --proof tptp tstp
Z3 n/a proof object n/a (set-option :produce-proofs true) + (get-proof) n/a

Notes:

  • When only proof objects or traces exist, populate ged_proof_graph or ged_trace_graph and leave ged_search_graph null.
  • Set trace_completeness to proxy when traces are not true search graphs.
